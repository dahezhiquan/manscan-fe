# 前端代码生成质量/格式检查清单

本文档用于前端开发、自测、联调和提交前的统一检查。目标不是增加流程负担，而是帮助我们在提交代码前尽早发现问题，保证功能质量、可维护性、性能、安全性、可访问性与文档完整性。

## 功能检查

- 所有功能都能正常工作
  - 新增功能、修改过的功能、受影响的关联功能都应实际验证。
- 边界情况都处理了
  - 包括空值、超长输入、重复点击、网络超时、权限不足、资源不存在、部分字段缺失等情况。
- 错误情况有友好提示
  - 错误信息应明确、可理解，避免只弹出技术性报错或静默失败。
- 页面状态完整
  - 加载中、空状态、失败状态、无权限状态、提交中状态都应有明确反馈。
- 用户体验流畅
  - 页面切换、表单提交、按钮交互、弹窗开关、失败重试等流程应自然顺畅，避免卡顿、阻塞或无反馈。
- 权限与可见性正确
  - 不同角色、不同环境、不同数据状态下的按钮、菜单、入口和操作范围应符合预期。

## 代码检查

- 代码符合项目规范
  - 保持与项目既定目录结构、编码风格、模块划分和工程约定一致。
- 变量、函数、组件命名清晰
  - 命名应准确表达意图，避免使用含义模糊的缩写、临时变量名或与职责不符的名称。
- 组件职责单一
  - 一个组件只解决一个主要问题，避免把数据请求、复杂业务逻辑和大段视图堆在同一个文件里。
- 没有重复代码
  - 相似逻辑优先抽取为公共组件、组合式函数、工具函数或常量，避免复制粘贴式实现。
- 复杂逻辑有注释
  - 仅在逻辑不直观、约束较多或存在隐含前提时补充简洁注释，重点说明“为什么这么做”。
- 没有被注释掉的代码
  - 废弃逻辑应直接删除，不保留大段注释代码。
- 没有魔法值
  - 重复出现的状态值、阈值、映射关系、路由名、权限码等应抽到常量或配置中统一维护。
- 类型和数据结构清晰
  - 接口数据、表单数据、组件参数、事件载荷等结构应明确，避免含义不清的自由拼装对象。

## 性能检查

- 没有不必要的重复渲染
  - 避免在模板、计算属性、侦听逻辑和高频交互中重复执行昂贵计算。
- 列表渲染使用了稳定唯一的 `key`
  - 禁止在可变列表中使用不稳定索引作为 `key`。
- 大数据量场景做了优化
  - 列表、表格、日志、树形结构等高数据量场景应考虑分页、分段加载、虚拟滚动或懒渲染。
- 请求触发时机合理
  - 避免重复请求、竞态请求和无意义的页面初始化请求。
- 静态资源经过优化
  - 图片、图标、字体等资源应控制体积，并根据场景选择合适格式、尺寸和加载策略。
- 首屏和交互性能可接受
  - 首屏渲染、搜索输入、筛选切换、滚动加载、弹窗打开等关键交互不应明显卡顿。

## 安全检查

- 用户输入都经过校验和转义处理
  - 对输入内容做合法性、长度、格式和必要的白名单校验，避免异常数据直接进入页面或请求链路。
- 不信任任何后端返回数据
  - 页面渲染前应校验字段存在性、类型和可用性，避免因接口异常导致页面报错。
- 禁止直接渲染不可信 HTML
  - 除非经过严格消毒，否则不要使用 `v-html` 渲染外部或用户输入内容。
- 敏感信息没有暴露
  - 不得在源码、日志、浏览器控制台、页面文案或打包产物中暴露密码、令牌、密钥等敏感内容。
- API 地址和密钥使用环境变量
  - 环境相关配置必须通过环境变量或安全配置注入，禁止硬编码到仓库中。
- 权限控制不能只做前端判断
  - 前端可以控制展示和交互，但不能把前端隐藏当作真正的权限校验。

## 测试检查

- 核心功能有测试
  - 关键业务流程、公共组件、复杂表单逻辑、数据转换逻辑应具备对应测试。
- 易回归问题有覆盖
  - 历史上容易出错的交互链路、条件分支和边界场景应补充自动化测试。
- 所有测试都通过
  - 提交前应确保与本次改动相关的自动化测试全部通过。
- 手动验证了主要流程
  - 对核心用户路径进行实际操作验证，尤其是自动化测试难以完整覆盖的交互链路。
- 跨端表现符合预期
  - 至少验证桌面端主流浏览器；如包含移动端页面，还应验证常见尺寸和触控交互。

## 文档检查

- README 是最新的
  - 功能说明、启动方式、依赖要求、构建命令、环境变量和使用说明应与当前实现保持一致。
- 重要组件和公共方法有注释
  - 对关键入口、公共组件、复杂核心逻辑补充必要注释，提升维护效率。
- 环境变量有说明
  - 需要的环境变量应写明名称、用途、默认值、是否必填及示例。
- 接口约定有文档
  - 前后端联调依赖的重要接口、字段约束、异常分支和特殊状态应有清晰说明。

## Frontend 开发规范

本节适用于前端源码目录。前端代码应优先保证可读性、可维护性和交互稳定性。除非当前任务明确要求，否则不要引入与现有工程体系冲突的新框架、新状态库或新样式体系。

### UI 设计前置要求

- 只要任务涉及前端页面、组件、交互、样式、布局、视觉呈现、体验优化或前端界面重构，生成代码前都必须先使用 `ui-ux-pro-max` skill。
- 这是硬性要求，不是可选建议。
- 禁止跳过 `ui-ux-pro-max` 直接开始写页面、组件或样式代码。
- 禁止凭空拍脑袋决定视觉风格、配色、字体、落地页结构、图表类型或交互细节。
- 即使需求描述很简单，只要产出的是前端 UI，也必须先通过 `ui-ux-pro-max` 获取设计方向。
- 如果任务同时包含前后端内容，只要其中有前端界面生成或修改，前端部分也必须先走 `ui-ux-pro-max`。

### ui-ux-pro-max 使用流程

- 第一步必须运行 `ui-ux-pro-max` 的 `--design-system` 查询，先产出整体设计方向，再开始写代码。
- 查询词必须至少包含以下信息中的大部分：
  - 产品类型，例如 `dashboard`、`landing page`、`admin`、`form`、`detail page`
  - 行业或场景，例如 `security`、`healthcare`、`fintech`、`education`
  - 风格关键词，例如 `professional`、`minimal`、`premium`、`playful`、`data-dense`
- 如果任务是多页面、多模块或中大型功能，必须使用 `--persist` 保存设计系统，作为后续页面实现的统一依据。
- 在完成 `--design-system` 之后，如有必要，再补充使用 `--domain` 或 `--stack` 查询细节。
- 当仓库技术栈明确时，前端实现前还必须补充一次对应 `--stack` 查询，获取框架级最佳实践。

### ui-ux-pro-max 产出如何落地到代码

- 页面结构必须参考 `PATTERN` 输出，不要随意改成另一套信息架构。
- 颜色、字体、主视觉风格必须参考 `COLORS`、`TYPOGRAPHY`、`STYLE` 输出，不要临时自由发挥。
- 动效、hover、focus、层次和质感处理必须参考 `KEY EFFECTS` 输出。
- 明确标记为 `AVOID` 的反模式禁止出现在最终代码中。
- 如果搜索结果与现有产品视觉体系冲突，应以“继承现有体系并吸收建议”为原则，而不是完全推翻已有设计语言。
- 如果第一次查询结果不理想，应调整关键词重新查询，禁止带着明显不合适的设计方向继续编码。

### 技术栈约定

- 默认使用 Vue 3 Composition API 编写组件逻辑。
- 工程构建基于 Vite。
- 优先使用 ES Module 组织代码。
- 除非项目已有明确约定，否则不要混用多套状态管理方案、多套请求封装方案和多套样式体系。
- 禁止使用 `alert`、`confirm`、`console.log` 作为正式交互或调试输出保留在提交代码中。

### 目录结构约定

- `src/pages`
  - 存放页面级组件，一个页面对应一个主要路由视图。
- `src/components`
  - 存放可复用组件，按业务域或通用能力分类。
- `src/composables`
  - 存放可复用的组合式逻辑，例如分页、表单状态、请求封装、权限判断。
- `src/api`
  - 存放接口定义和请求方法，不在组件中直接拼接请求细节。
- `src/stores`
  - 存放全局状态，仅保存跨页面、跨组件共享的状态。
- `src/router`
  - 存放路由配置、路由守卫和页面级导航逻辑。
- `src/utils`
  - 存放纯工具函数，不包含页面副作用和界面耦合逻辑。
- `src/constants`
  - 存放常量、枚举、状态映射和配置项。
- `src/assets`
  - 存放图片、图标、字体和其他静态资源。
- `src/styles`
  - 存放全局样式、主题变量、重置样式和通用样式工具。

### 分层职责约定

- 页面组件负责页面编排、路由参数处理和页面级状态组织。
- 业务组件负责具体交互和局部展示，不承担全局路由和全局状态管理职责。
- `composables` 负责抽取可复用的状态逻辑、副作用逻辑和业务流程控制。
- `api` 层只负责接口请求封装、参数传递和响应结构整理，不包含界面逻辑。
- `utils` 只负责纯函数逻辑，禁止依赖组件实例、DOM 或路由上下文。
- 全局状态只存放真正需要跨区域共享的数据，禁止把局部表单状态、弹窗开关等短生命周期状态滥用为全局状态。

### 组件开发约定

- 单文件组件应按 `template`、`script`、`style` 的顺序组织。
- 优先使用 `script setup`。
- 组件名使用大驼峰命名，例如 `UserTable.vue`、`AssetDetailDialog.vue`。
- 基础组件与业务组件要分层
  - 基础组件强调通用能力，业务组件强调具体业务语义，禁止混为一谈。
- `props`、`emits`、插槽能力应显式声明，避免隐式约定。
- 组件不能直接修改传入的 `props`。
- 复杂组件应拆分子组件，避免出现超大文件和多重职责耦合。
- 组件内部避免直接操作 DOM
  - 只有在焦点控制、测量尺寸、第三方库集成等必要场景下才允许，并应封装清晰。

### 状态管理约定

- 能用局部状态解决的问题，不要上升到全局状态。
- 共享状态必须有明确来源、更新入口和消费边界。
- 异步状态应明确区分 `idle`、`loading`、`success`、`error` 等阶段。
- 不要在多个地方重复维护同一份业务状态，避免状态漂移。
- 派生状态优先通过计算属性或纯函数生成，不要重复存储可推导数据。

### 接口与数据流约定

- 所有 HTTP 请求统一收敛到 `api` 层管理。
- 组件中禁止直接写死接口地址、请求头和错误处理分支。
- 请求方法名使用动词加资源语义命名，例如 `getUserList`、`createTask`、`updatePolicy`。
- 请求参数和返回数据应做结构化整理
  - 不要把原始接口响应对象在页面中到处透传。
- 页面展示前应完成字段兜底、空值兼容和必要的数据格式转换。
- 对于会取消、覆盖或重复触发的请求，应明确处理竞态关系。

### 路由约定

- 路由路径、路由名和页面文件命名应保持一致的语义规则。
- 路由守卫只处理登录态、权限、页面标题和少量导航前置逻辑。
- 不要把复杂业务流程塞进路由守卫。
- 页面级数据加载逻辑应尽量放在页面组件或对应组合式函数中维护。

### 表单与交互约定

- 表单项应具备默认值、校验规则、错误提示和提交中状态。
- 提交按钮在请求进行中应避免重复触发。
- 删除、覆盖、批量操作等高风险行为应提供明确确认。
- 成功反馈和失败反馈要可感知
  - 用户需要知道操作是否成功、失败原因以及下一步能做什么。
- 弹窗关闭、页面切换、刷新重试等操作不能导致重要用户输入无提示丢失。

### 样式约定

- 样式方案应保持统一，避免同时大量混用行内样式、全局样式、局部样式和临时覆盖。
- 颜色、字号、间距、圆角、阴影等设计 token 应尽量集中管理。
- 类名命名要有语义，避免大量使用 `box1`、`left-wrap`、`tmp` 之类的模糊名称。
- 页面样式优先跟随组件归属维护；全局样式只承载主题、重置和通用工具能力。
- 样式实现要兼顾响应式布局，避免只在单一屏幕尺寸下可用。
- 不要依赖过深的选择器嵌套或高权重覆盖来维持样式生效。

### 可访问性约定

- 表单控件应有明确标签或可感知名称。
- 图片、图标按钮、仅图形交互元素应提供必要的可访问文本。
- 键盘可达的交互必须可聚焦、可操作、可退出。
- 不要只依赖颜色表达状态，必要时应辅以文字或图标说明。
- 弹窗、抽屉、菜单等浮层组件应正确处理焦点和关闭行为。

### 命名与文件约定

- 组件文件使用大驼峰命名，例如 `UserFilterPanel.vue`。
- 组合式函数使用 `useXxx` 命名，例如 `usePagination`、`usePermission`。
- 工具函数文件使用小写或小写中划线风格，保持统一即可。
- 常量命名使用大写下划线或语义化对象映射，保持统一即可。
- 公开方法、导出函数和复杂组件建议配有简短注释，说明用途即可。

### 错误处理约定

- 所有异步请求都必须处理成功、失败和中断场景。
- 禁止吞掉异常
  - 捕获错误后必须记录、提示、转换或继续向上抛出，不能空 `catch`。
- 错误提示应对用户友好，对开发排查有信息量。
- 页面级错误、组件级错误和接口级错误应分层处理，避免所有问题都在最外层统一弹一个模糊提示。

### 代码生成与新增功能顺序

- 任何前端页面、组件、样式或交互代码生成前，必须先完成一次 `ui-ux-pro-max --design-system` 查询。
- 如果任务涉及具体框架落地，必须在编码前补充一次 `ui-ux-pro-max --stack <stack>` 查询。
- 生成代码时必须输出完整文件，包含完整依赖导入和完整组件结构。
- 不要省略上下文，不要只输出局部片段。
- 新增一个页面功能时，建议按以下顺序组织实现：
  - `ui-ux-pro-max --design-system`
  - `ui-ux-pro-max --stack`
  - `constants`
  - `api`
  - `composables`
  - `components`
  - `pages`
  - `router`
- 如果涉及全局共享状态，再补充 `stores`。
- 如果涉及通用能力沉淀，再补充 `utils` 或基础组件。

### Frontend 开发特别要求

- 生成前端代码时，务必务必务必要先使用 `ui-ux-pro-max` skill。
- 没有经过 `ui-ux-pro-max` 设计系统查询的前端代码，视为不符合规范。
- 前端代码不仅要“能用”，还要有明确设计方向，不能产出模板味、默认风格、随意拼接的界面。
- 尽量复用现有组件、组合式函数、请求封装和样式变量，避免重复造轮子。
- 优先写清晰、稳定、容易维护的代码，不为过度抽象牺牲可读性。
- 在没有明确收益的情况下，不要为了“高级写法”引入额外复杂度。
- 除非当前任务明确要求，否则不要顺手重构无关模块，不要扩大改动范围。
